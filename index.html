Code Review

This code review identifies weaknesses in the original CS 340 artifact (lack of authentication, unoptimized queries, absent access controls) and articulates the clear enhancement roadmap implemented above.

🔍

1. Existing Functionality

The original CS 340 artifact is a Python-based CRUD module using the Dash framework to interface with a MongoDB database. It provides basic operations (Create, Read, Update, Delete) to manage animal shelter data, serving as a functional backend prototype for a web dashboard.

⚠️

2. Code Analysis

A review of the original codebase revealed significant architectural weaknesses. There was no user authentication or input validation, leaving it vulnerable to injection. Database queries executed unindexed O(n) collection scans, and there was a complete absence of access controls, meaning any user could execute destructive commands.

🚀

To address these flaws, the roadmap focused on three categories: Software Engineering (implementing bcrypt authentication and strict input validation), Algorithms (engineering O(log n) B-Tree indexing and constant-time O(1) caching), and Databases (establishing role-based access control and automated forensic audit logging).

Artifact Reflections

Enhancement Narratives

Enhancement One: Software Design & Engineering

Artifact Description

The artifact selected for this category is the Animal Shelter CRUD application originally developed in CS 340: Advanced Programming Concepts. It is a Python-based web dashboard built with the Dash framework that interfaces with a MongoDB database to perform create, read, update, and delete operations on animal shelter data for the fictional client Grazioso Salvare.

Justification & Improvements

While functional, the original artifact lacked any authentication, input validation, or data protection mechanisms. I selected it to demonstrate my ability to take a prototype and elevate it to production-grade security standards.

The software engineering enhancement focused on adding security hardening to the application layer. This included implementing user authentication via hashed credentials (bcrypt), input validation on all user-supplied fields to prevent injection attacks, and encrypting sensitive data fields at rest. These enhancements showcase my skills in secure software design, defensive programming, and the application of cryptographic principles.

Course Outcomes Alignment

Through this enhancement, I successfully met the planned course outcomes by demonstrating the ability to use well-founded techniques in computing practices. By implementing proactive security measures against injection and authentication bypass attacks, I illustrated substantial progress toward developing a security mindset that anticipates adversarial exploits and mitigates design flaws.

Reflection & Challenges

The process of enhancing this artifact reinforced the critical importance of a defense-in-depth approach. One of the primary challenges was configuring the bcrypt library in Python and ensuring that password hashing integrated smoothly with the existing Dash callbacks without breaking the user experience. By resolving these issues, I deepened my practical understanding of integrating robust security mechanisms into existing codebases, directly aligning with my career goals as a secure software engineer.

Enhancement Two: Algorithms & Data Structures

Artifact Description

The artifact selected for this enhancement is the same Animal Shelter CRUD application. The underlying Python module handles all interactions with the MongoDB database. This artifact served as a functional prototype but suffered from severe algorithmic inefficiencies.

Justification & Improvements

I selected this artifact for the Algorithms category because it presented a clear, measurable performance problem that could be solved through direct algorithmic engineering. In its original state, every read operation executed a full MongoDB collection scan, resulting in an O(n) time complexity problem that would render the dashboard unusable at scale.

I engineered a MongoDB indexing strategy by explicitly calling `create_index()` for frequently queried fields (animal type, breed, outcome type). This allows MongoDB to utilize a B-Tree data structure, reducing search time to O(log n). Secondly, I implemented an in-memory caching layer using a Python hash map (dictionary). The `read()` method now serializes queries into JSON string keys, providing O(1) constant-time retrieval for cached queries. Finally, I implemented cache invalidation logic to clear the cache during any write operations, ensuring data integrity.

Course Outcomes Alignment

I demonstrated the ability to design computing solutions that solve a problem using algorithmic principles, while managing the trade-offs involved in design choices. I analyzed a performance bottleneck, selected a B-Tree index for search efficiency and a hash map for constant-time retrieval, and carefully navigated the trade-off of cache invalidation overhead versus read speed optimization.

Reflection & Challenges

A major challenge was designing a consistent cache key, since MongoDB query objects are dictionaries and not directly hashable. I resolved this by serializing the query to a JSON string with sorted keys, which produces a deterministic string for any query. This enhancement reinforced that algorithmic thinking is deeply embedded in everyday software engineering decisions, turning theoretical time complexity concepts into actionable performance fixes.

Enhancement Three: Databases

Artifact Description

This final enhancement targets the database layer of the Animal Shelter CRUD application. The module interacts directly with MongoDB to power the frontend visualization tools.

Justification & Improvements

In its original implementation, the application utilized a single MongoDB connection string with broad, unrestricted read/write permissions. Any user had full administrative power, and there was absolutely no accountability mechanism. To transform this into a production-grade system, I implemented Role-Based Access Control (RBAC) and automated Audit Logging.

I created an authentication layer that dynamically maps users to 'admin', 'staff', or 'viewer' roles, enforcing the principle of least privilege across all operations (e.g., restricting 'delete' entirely to admins). Secondly, I engineered a `_log_action()` method that automatically inserts a forensic record into a dedicated 'audit_log' MongoDB collection whenever a destructive action occurs, capturing the timestamp, user role, action, and affected document.

Course Outcomes Alignment

This directly satisfied the outcome of developing a security mindset that anticipates adversarial exploits and ensures privacy and enhanced security of data. Furthermore, structuring database permissions around organizational roles (admin, staff, viewer) demonstrated the ability to employ strategies for building collaborative environments that support organizational decision-making safely.

Reflection & Challenges

A significant challenge was engineering the audit log so it could not be bypassed. I deliberately implemented the logging logic deeply within the core Python CRUD module rather than the frontend, ensuring that no matter what interface calls the database, the forensic trail is always generated. This process profoundly deepened my understanding of defense-in-depth database architecture.
